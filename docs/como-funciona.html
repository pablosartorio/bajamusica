Sonido cómo funciona

Cómo funciona todo el sistema

Sonido es un descargador de YouTube con interfaz web local. Buscás o pegás una playlist, elegís qué bajar y lo guardás como MP3/MP4. Este documento explica la arquitectura, el ciclo de una descarga y el empaquetado de Windows.

1 Vista general

Una sola página servida por Flask en 127.0.0.1:5000. Nada sale a internet salvo las requests a YouTube (vía yt-dlp) y, opcionalmente, a MusicBrainz para enriquecer metadatos. La conversión de audio/video la hace ffmpeg localmente.

3 piezas

Un servidor Flask fino, un paquete core con toda la lógica, y un frontend vanilla (HTML+CSS+JS, sin frameworks ni build).

Sin estado en disco… casi

El estado de las tareas vive en memoria; lo único que se persiste es el historial de descargas completadas en un JSON.

2 Arquitectura

El backend está desacoplado del frontend: el paquete core no sabe nada de Flask. Las rutas en app.py son finas — reciben, validan y delegan.

3 Flujo de uso

De punta a punta, lo que pasa cuando el usuario baja música:

1 · buscar
Escribe o pega URL
El frontend detecta si es playlist (?list=) y enruta a /expand_playlist; si no, a /search.
↓
2 · elegir
Tilda temas + opciones
Formato (MP3/MP4), calidad, esquema de nombre y carpeta destino (/browse_dir abre el diálogo nativo).
↓
3 · descargar
POST /download
El server crea un job_id y lanza un thread de descarga. Responde el id al instante.
↓
4 · seguir
GET /progress/<job_id>
El frontend pollea cada 500 ms y pinta el panel (deck) con barras VU por archivo + barra global.
↓
5 · guardar
Historial
Al terminar, el job se persiste en history.json. El botón Historial lo muestra.
Polling, no WebSockets: para un caso local, pollear /progress es más simple y robusto. Cerrar el panel no detiene el seguimiento: la descarga sigue en el servidor y se guarda igual.

Validación de /download: rechaza con 400 si no hay items, si ningún item trae id válido, o si el formato no está soportado. La calidad inválida cae a "high".

5 Ciclo de una descarga

Cada item de una tarea recorre estos estados (se procesan secuencialmente). El frontend los mapea a las fases que ves en el panel:

queued→ downloading→ converting→ tagging*→ done   |  cualquier fallo → error

* tagging solo ocurre si el esquema de nombre no es "youtube": ahí se consulta MusicBrainz, se escriben las etiquetas ID3 y se renombra el archivo.

Robustez: el job se cierra siempre al final (set_overall("done") en un finally). Si la carpeta destino es inválida o sin permisos, todos los items se marcan error en vez de dejar la UI polleando para siempre.

Cómo llega el progreso

  • yt-dlp → hook: mientras baja, un progress hook reporta bytes, velocidad y ETA a jobs.update_item.
  • jobs.py guarda el estado en un dict en memoria, protegido por un Lock.
  • El frontend pide /progress/<id> cada 500 ms y vuelve a pintar el panel.
  • Al terminar, set_overall("done") persiste el job en el historial.

6 Backend · paquete core

search.py

Busca en YouTube con yt-dlp en modo flat (rápido, sin resolver streams). Si la query es una URL de video, la resuelve directo.

core/search.py

playlist.py

Expande una playlist completa a sus entradas (extract_flat="in_playlist"), devolviendo título + lista de videos.

core/playlist.py

download.py

El corazón: arma las opciones de yt-dlp según formato/calidad, descarga, dispara ffmpeg para convertir y reporta progreso.

core/download.py

jobs.py

Registro thread-safe del estado de cada tarea (en memoria) y persistencia del historial en disco (máx. 500 entradas).

core/jobs.py

metadata.py

Limpia el título de YouTube, busca en MusicBrainz, escribe etiquetas ID3 (MP3) y renombra el archivo. Todo best-effort: nunca rompe la descarga.

core/metadata.py

util.py

Helpers reutilizables, como formatear segundos a m:ss / h:mm:ss.

core/util.py

8 Empaquetado Windows (.exe)

build.bat genera un ejecutable autónomo con PyInstaller (modo onedir), bundleando Python, las dependencias y ffmpeg. La PC destino no necesita instalar nada.

# build.bat hace, en orden:
1. crea un venv de build aislado
2. instala dependencias + PyInstaller
3. baja ffmpeg.exe / ffprobe.exe  (_get_ffmpeg.py)
4. arma el bundle             (bajamusica.spec)
   → dist\bajamusica\bajamusica.exe
La trampa de PyInstaller 6.x: en modo onedir, los binarios y datos NO quedan al lado del .exe sino dentro de una subcarpeta _internal\. Por eso las rutas se resuelven con sys._MEIPASS (no con sys.executable):
  • ffmpeg → config.FFMPEG_LOCATION busca el binario en _MEIPASS.
  • templates / static → app._ROOT también sale de _MEIPASS.
Si no, yt-dlp no encuentra ffmpeg y los archivos se bajan pero no se convierten a MP3.

9 Datos & configuración

  • Config central: todo lo ajustable (puerto, carpeta destino, calidades, límite de búsqueda, ruta del historial) vive en config.py.
  • Carpeta destino por defecto: ~/Music/YT-Downloads (en Windows, C:\Users\…\Music\YT-Downloads). Se puede cambiar desde la UI.
  • Historial: Linux → ~/.local/share/bajamusica/history.json · Windows → %APPDATA%\bajamusica\history.json. Máximo 500 entradas.
  • Búsqueda: hasta MAX_SEARCH_RESULTS = 30 resultados.
